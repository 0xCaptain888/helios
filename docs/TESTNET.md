# Casper Testnet Runbook

Everything needed to take Helios from the local simulation to **live
transactions on Casper Testnet** — the qualification requirement.

## 0. Prerequisites

| Tool | Install |
|---|---|
| Rust + wasm target | `rustup target add wasm32-unknown-unknown` |
| `casper-client` | **NOT required** — pure Python deployer replaces it |
| Python 3.10+ | already on your machine |
| `cryptography` pip package | `pip install cryptography` (one package only) |
| 5 funded testnet accounts | auto-generated by the deploy script |

> **casper-client is completely optional.** All deployment and contract calls
> are handled by `scripts/casper_deploy.py` which uses Python + the
> `cryptography` package for ed25519 signing and speaks directly to the
> Casper JSON-RPC endpoint. If you already have casper-client working, you
> can still use it — but you don't have to.

## 1. Build the contracts

```bash
# DO NOT use `cargo odra build` — see BUILD.md and Trap 5 in troubleshooting.
# Use the plain cargo build script instead:
bash scripts/build_contracts.sh
# Expected output: 4x OK from check_wasm_exports.py
```

## 2. Deploy — one command (recommended)

```bash
pip install cryptography          # only dependency
python3 scripts/deploy_helios.py  # generates keys, prompts for faucet, deploys all 4 contracts
```

The script handles everything:
- generates 5 keypairs in `keys/`
- prints faucet URLs and waits for you to fund them
- deploys OracleRegistry → DataMarket → wires them → FundVault → Governance
- writes `agents/testnet.env` automatically
- produces first on-chain activity (register + attest)

Resume after interruption: `python3 scripts/deploy_helios.py --resume agents/testnet.env`

**Manual alternative** (if you prefer step-by-step or want to use casper-client):

```bash
python3 scripts/casper_deploy.py install --key keys/fund_agent/secret_key.pem \
  --wasm contracts/wasm/OracleRegistry.wasm --wait

python3 scripts/casper_deploy.py call --key keys/fund_agent/secret_key.pem \
  --contract <REGISTRY_HASH> --entry-point register \
  --args "name:string=TBill Oracle" "category:string=rwa" \
         "endpoint:string=http://localhost:8451/quote" "price_motes:u64=2000000000"
```

Order matters:

1. **OracleRegistry** (no init args beyond defaults)
2. **DataMarket** — init args: `registry` (OracleRegistry contract hash), `fee_bps = 250`
3. wire them: call `set_market(market_hash)` on OracleRegistry **from the deployer/admin key**
4. **FundVault** — init arg: `operator` = fund_agent's account hash
5. **Governance** — init args: `fund_agent`, `risk_agent` account hashes, `veto_window_ms` (e.g. `60000`)

Example deploy command (repeat per WASM, adjust args):

```bash
casper-client put-txn session \
  --node-address https://node.testnet.casper.network/rpc \
  --chain-name casper-test \
  --secret-key keys/fund_agent/secret_key.pem \
  --wasm-path contracts/wasm/OracleRegistry.wasm \
  --payment-amount 300000000000 \
  --gas-price-tolerance 1 \
  --standard-payment true \
  --install-upgrade
```

Record each resulting contract hash:

```bash
casper-client get-txn --node-address https://node.testnet.casper.network/rpc <TX_HASH>
# or look the deploy up on https://testnet.cspr.live
```

> Note: `casper-client` syntax differs between 1.x (`put-deploy`) and 2.x
> (`put-txn`). The commands here target 2.x / Casper 2.0. If your client says
> `put-txn` is unknown, upgrade the client or substitute the equivalent
> `put-deploy --session-path ...` form.

## 3. Configure the agents

Create `agents/testnet.env` (gitignored):

```ini
ORACLE_TBILL_KEY=keys/oracle_tbill/secret_key.pem
ORACLE_GOLD_KEY=keys/oracle_gold/secret_key.pem
ORACLE_REINDEX_KEY=keys/oracle_reindex/secret_key.pem
FUND_AGENT_KEY=keys/fund_agent/secret_key.pem
RISK_AGENT_KEY=keys/risk_agent/secret_key.pem
REGISTRY_HASH=entity-contract-XXXX...
MARKET_HASH=entity-contract-XXXX...
VAULT_HASH=entity-contract-XXXX...
GOV_HASH=entity-contract-XXXX...
```

## 4. Produce on-chain activity

```bash
HELIOS_MODE=testnet python3 scripts/testnet_round.py --register --list   # one-time onboarding
HELIOS_MODE=testnet python3 scripts/testnet_round.py --rounds 3          # attest → anchor → propose → finalize → rebalance
```

Each round produces ~10 testnet transactions. Verify on
`https://testnet.cspr.live/account/<account-hash>` and link those URLs in the
submission.

## 5. Reading state for the dashboard

The dashboard's feed is written locally by the agents; for judges who want to
verify directly against the chain:

```bash
casper-client query-global-state \
  --node-address https://node.testnet.casper.network/rpc \
  --state-root-hash <SRH> \
  --key entity-contract-<REGISTRY_HASH> -q "oracles"
```

or use CSPR.cloud's REST API with a free API key for indexed queries.

## Troubleshooting

| Symptom | Fix |
|---|---|
| `Wasm preprocessing error: Memory section should exist` | Built with `--import-memory`. Run: `unset RUSTFLAGS && bash scripts/build_contracts.sh`. Check `~/.cargo/config.toml` for the flag too. |
| `Out of gas` on deploy | raise `--payment-amount` (WASM deploys on testnet often need 200–400 CSPR) |
| `invalid deploy: account has insufficient balance` | hit the faucet again; each account should hold ≥ 1000 test CSPR before deploying |
| `cargo odra build` fails on Odra version | this repo pins `odra = 1.4.0`; if your cargo-odra is newer, either `cargo install cargo-odra --version 1.4.0 --locked` or bump the `odra` version in `contracts/Cargo.toml` and re-run tests — entry-point code uses stable Odra APIs |
| agents can't find casper-client | ensure `~/.cargo/bin` is on PATH |

## The three known build traps (verified in the field)

These three issues form a chain — compile-time, dependency-resolution, and
artifact-level — and each one hides the next. Fix them in order.

### Trap 1 — `#[no_mangle]` cannot be used on internal language items

`odra-casper-wasm-env` 1.4.0 puts both `#[no_mangle]` and `#[panic_handler]`
on its panic fn; modern rustc rejects this as a hard error.

**Fix (reproducible — do NOT hand-edit `~/.cargo/registry`, checksums will
break and the fix evaporates on any other machine):** vendor the source and
patch it in-repo:

```bash
cd contracts
git clone --depth 1 --branch v1.4.0 https://github.com/odradev/odra vendor/odra
# delete the `#[no_mangle]` line above the panic handler in
# vendor/odra/odra-casper/wasm-env/src/
```

Then uncomment the `[patch.crates-io]` block at the bottom of
`contracts/Cargo.toml` and commit `vendor/` — the patch now travels with the
repo (CI, judges, your other machines).

*Alternative:* pin an older toolchain via `contracts/rust-toolchain.toml`
(`channel = "<your last-known-good rustc>"`), which avoids source surgery but
forces Trap 2's fix.

### Trap 2 — `feature edition2024 is required` (dependency deadlock)

Older toolchains + fresh dependency resolution = Cargo pulls e.g.
`base64ct v1.8+` built on edition 2024, which old Cargo can't parse.

**Fix:**

```bash
cargo update -p base64ct --precise 1.6.0
```

and **commit `Cargo.lock`** (this repo's `.gitignore` intentionally does NOT
exclude it). The lock file freezes the verified dependency graph so CI and
anyone cloning the repo can never re-trigger the deadlock.

### Trap 3 — wasm builds "successfully" but only exports `main`, on-chain fails with `Module doesn't have export call`

When `Cargo.toml` has `[[bin]]` targets and your `cargo-odra` version doesn't
match odra 1.4's build flow, it compiles the binary (exporting `main`) instead
of the contract lib (which must export `call`). The build *looks* green — the
failure only surfaces on-chain after burning gas.

**Fix, in order of preference:**

1. Match the toolchain: `cargo odra --version`, then
   `cargo install cargo-odra --version <the 1.4-era release> --locked`.
2. Or, if staying on a newer cargo-odra: comment out both `[[bin]]` sections
   in `contracts/Cargo.toml` and drop the `odra-build` build-dependency, so
   the tool builds the lib through its own flow.
3. **Always run the gate before deploying** (the deploy script now enforces
   this automatically and refuses to submit a broken wasm):

```bash
python3 scripts/check_wasm_exports.py contracts/wasm/*.wasm
# OK    ...OracleRegistry.wasm: exports [call]   <- required for every contract
```

The checker is zero-dependency (pure Python, no wabt needed) and parses the
wasm export section directly.

### Trap 4 — `warning: patch ... was not used in the crate graph` (the silent killer)

Cargo's `[patch]` only engages when the patched crate's version **exactly
equals** the version the resolver picked. `version = "1.4.0"` in Cargo.toml
means `^1.4.0` (caret semantics) — so once odra published 1.5.1, cargo
resolved 1.5.1, found the 1.4.0 patch version-mismatched, and **silently
ignored it**, recompiling the unpatched registry copy forever. The build error
never changes no matter what you edit locally, because your local code is not
in the build.

**Fix (automated by `scripts/apply_odra_patch.sh` v2):**

1. Pin exact versions: `odra = { version = "=1.5.1" }` (same for odra-test /
   odra-build) — `=` disables caret drift.
2. Vendor the **matching** tag (`v1.5.1`) and apply the panic-handler patch
   to it.
3. **Never trust the warning-free build alone — verify engagement:**

```bash
cargo tree -i odra-casper-wasm-env
# MUST print a path into vendor/, e.g.:
#   odra-casper-wasm-env v1.5.1 (.../contracts/vendor/odra/odra-casper/wasm-env)
# If it prints (registry+https://github.com/rust-lang/crates.io-index) — the
# patch is NOT in use; the shown version tells you what to re-target.
```

The v2 script performs this verification automatically and exits non-zero if
the patch did not take over.

### Trap 5 — `odra-schema` silently pulls `odra-core >=1.5.1`; the zsh glob bomb hides it all

**What happens:** Adding `[[bin]]` pulls `odra-build` → `odra-schema`.
`odra-schema` on crates.io requires `odra-core >=1.5.1`. If your local patch
supplies `1.4.0`, Cargo refuses the whole graph. And because `cargo odra build`
internally runs `rm -rf wasm/*.wasm && cargo build …`, zsh kills the entire
command on an empty-directory glob *before* cargo ever runs — so you keep
seeing the same error and never see a new one.

**Fix: no `cargo odra` at all.** This repo now compiles with plain cargo via
feature flags:

```bash
# builds OracleRegistry.wasm (exports `call`, not `main`)
cargo build --release --target wasm32-unknown-unknown \
  --features oracle-registry --no-default-features

# or use the safe build script (no globs, works in zsh/bash):
bash scripts/build_contracts.sh
```

No `[[bin]]`, no `odra-build`, no `odra-schema`, no version-conflict chain,
no zsh glob bomb. Each feature flag gates exactly one `#[no_mangle] pub
extern "C" fn call()`, so the wasm always exports `call`.
